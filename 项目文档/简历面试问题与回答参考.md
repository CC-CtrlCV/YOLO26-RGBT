# 简历面试问题与回答参考

本文档用于准备将本项目写入简历后的面试问答。

项目名称可以写成：

```text
基于 Ultralytics YOLO 的 RGB + 红外多模态目标检测系统
```

简历描述可以写成：

```text
基于 Ultralytics YOLO 框架完成 RGB + 红外双模态目标检测适配，支持 RGBT 图像配对读取、4 通道输入、early fusion、mid fusion、训练、验证、推理和模型导出，并适配 YOLOv5、YOLOv8、YOLOv11、YOLOv26 多版本模型配置。
```

## 1. 项目整体类问题

### 1.1 你这个项目主要做了什么？

参考回答：

本项目是在 Ultralytics YOLO 框架基础上做的 RGB + 红外多模态目标检测改造。原版 YOLO 默认读取 3 通道 RGB 图像，我在数据加载、模型输入、网络结构和推理流程中增加了 RGBT 支持，使项目可以自动读取可见光图像和对应红外图像，并拼接成 4 通道输入进行训练和推理。同时我提供了 early fusion 和 mid fusion 两类多模态模型配置，并整理了训练、验证、推理、导出脚本和中文说明文档，方便作为开源项目使用。

### 1.2 为什么要做 RGB + 红外多模态检测？

参考回答：

RGB 图像包含颜色、纹理、边缘等视觉信息，但在夜间、弱光、强逆光、烟雾等场景下容易退化。红外图像反映目标热辐射信息，对夜间行人、车辆等目标更敏感。RGB + 红外融合可以让模型同时利用可见光的纹理信息和红外的热信息，从而提高复杂环境下的检测鲁棒性。

### 1.3 这个项目和普通 YOLO 项目相比有什么不同？

参考回答：

普通 YOLO 项目通常输入单张 3 通道 RGB 图像，而本项目输入 RGB 图像和红外图像组成的图像对。主要区别有三点：第一，数据加载器会根据 visible 路径自动找到 infrared 图像；第二，模型输入从 3 通道扩展为 4 通道；第三，模型结构增加了 early fusion 和 mid fusion 多模态融合配置。

### 1.4 你在项目里主要负责了哪些工作？

参考回答：

我主要完成了四部分工作：第一，修改数据读取流程，支持 RGB 和红外图像自动配对并拼接为 4 通道；第二，修改模型解析逻辑，新增 ChannelSlice 模块，使模型 YAML 可以切分 RGB 和红外通道；第三，适配 YOLOv5、YOLOv8、YOLOv11、YOLOv26 的 RGBT 模型配置；第四，补充训练、验证、推理、导出脚本和开源文档，使项目可以完整运行和发布。

## 2. 数据集与数据读取问题

### 2.1 RGBT 数据集目录应该是什么样的？

参考回答：

推荐使用 visible、infrared、labels 三个目录。visible 中放 RGB 图像，infrared 中放红外图像，labels 中放 YOLO 格式标签。visible 和 infrared 下的文件名要一一对应，例如 `visible/train/000001.jpg` 对应 `infrared/train/000001.jpg`，标签为 `labels/train/000001.txt` 或与可见光图像同目录的同名 txt。

### 2.2 代码怎么找到红外图像？

参考回答：

数据集 YAML 中有 `rgbt_pair: [visible, infrared]`。训练时传入的是 visible 图像路径，代码会把路径中的 visible 替换成 infrared，从而找到对应红外图像。例如 `visible/train/000001.jpg` 会映射到 `infrared/train/000001.jpg`。

### 2.3 为什么数据集 YAML 里要写 channels: 4？

参考回答：

因为 RGB 图像是 3 通道，红外图像通常作为 1 通道灰度图，拼接后输入是 4 通道。`channels: 4` 用于告诉模型构建阶段第一层输入通道数应该是 4，而不是普通 YOLO 默认的 3。

### 2.4 如果 RGB 和红外图像尺寸不同怎么办？

参考回答：

代码中可以在读取后将红外图像 resize 到 RGB 图像尺寸，但更推荐在数据预处理阶段完成图像配准和尺寸统一。因为多模态检测依赖两个模态的空间对齐，如果 RGB 和红外没有配准，即使尺寸一致，也可能导致目标位置不一致，影响融合效果。

### 2.5 红外图像必须是灰度图吗？

参考回答：

本项目推荐使用单通道灰度红外图，因为最后拼接为 RGB 三通道加红外一通道。如果红外图是伪彩色图，建议先转为灰度图，否则会变成更多通道，模型结构也要重新设计。

## 3. 模型结构问题

### 3.1 early fusion 是什么？

参考回答：

early fusion 是早期融合，指在输入阶段就把 RGB 和红外图像拼接成 4 通道，然后送入同一个 YOLO backbone。它实现简单，训练流程接近原版 YOLO，适合作为 baseline。

### 3.2 mid fusion 是什么？

参考回答：

mid fusion 是中期融合，指先把 RGB 和红外分成两个分支，分别进行特征提取，然后在中间层或多尺度特征层进行融合。本项目通过 ChannelSlice 把 4 通道输入切成 RGB 三通道和红外一通道，再分别进入两个分支，最后用 Concat 进行特征融合。

### 3.3 early fusion 和 mid fusion 怎么选？

参考回答：

如果只是想快速跑通流程，early fusion 更合适，因为结构简单、显存占用相对较小。如果要做正式的多模态实验或论文改进，mid fusion 更合适，因为它显式保留了 RGB 和红外两个分支，更容易分析不同模态的作用，也方便后续加入注意力融合、加权融合等模块。

### 3.4 ChannelSlice 是什么作用？

参考回答：

ChannelSlice 是我新增的一个通道切分模块，用于从 4 通道 RGBT 输入中切出不同模态。例如 `ChannelSlice [0, 3]` 表示取 RGB 三通道，`ChannelSlice [3, 4]` 表示取红外一通道。它主要服务于 mid fusion 双分支结构。

### 3.5 为什么不能直接用原版 yolo26.yaml 训练 RGBT？

参考回答：

原版 yolo26.yaml 默认是 3 通道输入，没有 `channels: 4`，也没有 RGB/IR 分支和融合结构。如果直接用 4 通道数据训练，可能出现输入通道不匹配。即使只在原版 YAML 中加 `channels: 4`，它也只是一个简单 4 通道输入模型，没有显式多模态融合设计。

### 3.6 如果我一定想用原版 yolo26 结构做多模态，怎么做？

参考回答：

可以复制一份 yolo26.yaml，然后在顶部加 `channels: 4`，这样第一层卷积会按 4 通道构建。这种方法本质上是简单 early fusion baseline，结构是 `RGB + IR -> 4 通道 -> 原版 YOLO26`，但不属于真正双分支融合模型。

## 4. 训练与推理问题

### 4.1 训练命令中哪些参数最重要？

参考回答：

最重要的是 `--model`、`--data`、`--imgsz`、`--epochs`、`--batch` 和 `--device`。`--model` 指定 RGBT 模型 YAML，`--data` 指定数据集 YAML，`--imgsz` 是输入尺寸，`--epochs` 是训练轮数，`--batch` 影响显存占用，`--device` 指定 GPU 或 CPU。

### 4.2 Windows 下为什么不能用反斜杠换行？

参考回答：

反斜杠换行是 Linux/macOS shell 的写法，Windows PowerShell 中应该使用反引号换行，或者直接写成一行命令。因此项目文档里同时给了 PowerShell、多行和单行命令，避免新手复制命令时报错。

### 4.3 推理时为什么只传 visible 目录？

参考回答：

因为代码会以 visible 图像为入口，根据 `rgbt_pair` 自动找到 infrared 图像。用户只需要传入 visible 目录，代码内部会完成红外图像配对读取和 4 通道拼接。

### 4.4 训练结果怎么看？

参考回答：

训练结果保存在 `runs/rgbt/实验名称` 下，主要看 `results.png` 中 loss 是否下降、mAP 是否上升，也可以看 `train_batch*.jpg` 和 `val_batch*.jpg` 判断标签和预测框是否正常。最终权重主要看 `weights/best.pt`。

### 4.5 显存不足怎么办？

参考回答：

可以优先减小 batch size，比如从 16 降到 8、4 或 2；也可以减小输入尺寸 imgsz，比如从 640 降到 512；如果使用 mid fusion 显存仍然不足，可以先使用 early fusion 模型。

## 5. 算法技术面试问题

### 5.1 YOLO 是一阶段检测器还是两阶段检测器？

参考回答：

YOLO 是一阶段目标检测器，它直接在特征图上预测目标类别和边界框，不像 Faster R-CNN 那样先生成候选区域再分类回归。因此 YOLO 通常推理速度较快，适合实时检测场景。

### 5.2 Backbone、Neck、Head 分别是什么？

参考回答：

Backbone 负责从输入图像中提取不同层次的特征；Neck 负责多尺度特征融合，让模型同时利用浅层细节和深层语义；Head 负责输出最终的分类和边界框预测。本项目主要在输入和 backbone/中间特征融合层做 RGBT 适配。

### 5.3 多尺度特征融合为什么重要？

参考回答：

目标检测中不同目标尺度差异很大，小目标更依赖浅层高分辨率特征，大目标更依赖深层语义特征。多尺度融合可以同时兼顾小目标和大目标检测，提高检测鲁棒性。

### 5.4 mAP 是什么？

参考回答：

mAP 是 mean Average Precision，表示各类别 AP 的平均值。AP 是 Precision-Recall 曲线下的面积。目标检测中常见指标有 mAP50 和 mAP50-95，mAP50 表示 IoU 阈值为 0.5 时的平均精度，mAP50-95 表示 IoU 从 0.5 到 0.95 多个阈值下的平均结果，更严格。

### 5.5 NMS 是什么？

参考回答：

NMS 是非极大值抑制，用于去除重复检测框。它会按照置信度排序，保留高置信度框，并删除与其 IoU 过高的其他框。这样可以避免同一个目标被多个框重复检测。

### 5.6 为什么红外可以提升夜间检测？

参考回答：

夜间 RGB 图像受光照影响大，目标纹理和颜色信息会弱化。而红外图像主要反映热辐射，对行人、车辆等具有热特征的目标更稳定，所以在夜间和弱光环境下可以提供互补信息。

## 6. 多模态领域问题

### 6.1 多模态融合有哪些常见方式？

参考回答：

常见方式包括输入级融合、特征级融合和决策级融合。输入级融合就是 early fusion，在输入阶段拼接；特征级融合就是 mid fusion，在 backbone 或 neck 的特征层融合；决策级融合是 late fusion，两个模型分别预测后再融合结果。本项目实现了 early fusion 和 mid fusion。

### 6.2 多模态检测的难点是什么？

参考回答：

主要难点包括模态配准、数据同步、模态噪声、不同模态信息不均衡、融合策略设计和计算成本增加。如果 RGB 和红外没有空间对齐，融合反而可能带来干扰。

### 6.3 RGB 和红外哪个更重要？

参考回答：

这取决于场景。白天或光照充足场景 RGB 通常更丰富，夜间或弱光场景红外更稳定。多模态的价值在于互补，而不是简单认为某一个模态永远更重要。

### 6.4 如果某一模态缺失怎么办？

参考回答：

当前项目默认 RGB 和红外都存在，如果某一模态缺失会报错。工程上可以做模态缺失处理，例如用零图像、复制 RGB 灰度图、单模态 fallback 模型，或者训练时加入模态 dropout，提高模型对缺失模态的鲁棒性。

### 6.5 mid fusion 后还能怎么改进？

参考回答：

可以把简单 Concat 替换成注意力融合、SE/CBAM 模块、门控融合、跨模态 Transformer、动态权重融合等方法。也可以在 P3、P4、P5 多尺度特征上分别设计融合模块，再做消融实验比较效果。

## 7. 工程落地问题

### 7.1 这个项目怎么适配自己的数据集？

参考回答：

用户需要把数据整理成 visible、infrared 和 labels 结构，保证 RGB 和红外文件名一致，然后复制 `rgbt-template.yaml`，修改 path、train、val 和 names。如果目录名不是 visible/infrared，需要同步修改 `rgbt_pair`。

### 7.2 你怎么保证 RGB 和红外一一对应？

参考回答：

我提供了 `check_rgbt_pairs.py` 脚本，它会遍历 visible 目录中的图像，并根据 `rgbt_pair` 生成对应 infrared 路径，检查文件是否存在。如果缺失，会打印缺失配对，训练前可以先用这个脚本检查数据完整性。

### 7.3 这个项目发布成开源项目做了哪些整理？

参考回答：

我把训练、验证、推理、导出脚本放到了根目录，方便新手直接运行；补充了 requirements、README、操作说明、项目结构说明、网络结构说明和 GitHub 发布清单；同时整理了 .gitignore，避免提交 runs、权重文件和本地配置。

### 7.4 为什么要把脚本放在根目录？

参考回答：

开源项目面向的不只是熟悉源码的人，很多新手不愿意深入源码目录找入口。把 `train_rgbt.py`、`val_rgbt.py`、`predict_rgbt.py` 放在根目录，可以降低使用门槛，让用户一眼知道从哪里开始。

### 7.5 项目目前还有哪些不足？

参考回答：

目前主要实现了 RGBT 数据读取和 YOLO 多模态模型配置，融合方式以 Concat 为主，后续可以加入更复杂的注意力融合模块；另外目前主要支持图像级 RGBT 推理，视频级 RGBT 推理和模态缺失鲁棒性还可以进一步完善；还可以补充更多数据集 benchmark。

## 8. 开源项目相关问题

### 8.1 为什么要写这么多中文文档？

参考回答：

因为这个项目希望对新手友好，尤其是很多人第一次做多模态检测时，不清楚数据集怎么放、YAML 怎么写、Windows 命令怎么运行、训练结果怎么看。文档可以降低使用成本，也能减少重复答疑。

### 8.2 GitHub 发布前需要注意什么？

参考回答：

需要避免提交本地数据集、训练结果、权重文件和运行时配置，比如 `runs/`、`*.pt`、`ultralytics/settings.json`。还要把 YAML 中的本地绝对路径改成示例路径，并确保 README 中的命令能直接复现。

### 8.3 这个项目和直接 fork Ultralytics 有什么区别？

参考回答：

它不是简单 fork，而是在 Ultralytics 基础上增加了 RGBT 多模态能力，包括数据配对读取、4 通道输入、ChannelSlice 模块、RGBT 模型 YAML、训练验证推理脚本和中文教程，目标更聚焦于 RGB + 红外目标检测。

## 9. 可能被追问的细节

### 9.1 你修改了哪些核心源码？

参考回答：

主要修改了数据和模型两条线。数据方面包括 `ultralytics/data/base.py`、`dataset.py`、`loaders.py`、`build.py` 和 `engine/predictor.py`；模型方面包括 `ultralytics/nn/modules/conv.py`、`modules/__init__.py` 和 `nn/tasks.py`；此外新增了 RGBT 数据集 YAML、模型 YAML 和根目录脚本。

### 9.2 为什么推理也要改 loaders.py？

参考回答：

训练和推理使用的数据入口不完全一样。训练走 Dataset，推理会走 LoadImagesAndVideos 等加载器。如果只改训练数据集，训练能跑，但推理时仍然只能读取普通 RGB 图像。因此推理加载器也需要支持 RGB/红外配对读取。

### 9.3 预训练权重怎么处理？

参考回答：

普通 YOLO 预训练权重第一层通常是 3 通道，而 RGBT 模型是 4 通道，因此第一层权重尺寸可能不匹配。最稳的方式是从 YAML 从头训练。如果要加载 RGB 预训练权重，可以跳过第一层，或者将 RGB 权重复制/平均初始化红外通道。

### 9.4 为什么 mid fusion 参数量会变大？

参考回答：

因为 mid fusion 中 RGB 和红外有相对独立的分支，等于复制了部分 backbone 结构，因此参数量和计算量会增加。它的优势是模态特征表达更清晰，劣势是训练成本更高。

### 9.5 如果面试官问效果提升多少怎么办？

参考回答：

如果还没有完整 benchmark，不要虚报。可以回答：目前项目重点完成了工程适配和可复现实验流程，已经支持 RGBT 训练、验证和推理；具体提升需要在统一数据集、统一训练策略下与 RGB-only、IR-only、early fusion、mid fusion 做对比实验，后续会补充 KAIST、LLVIP 或 FLIR 上的 benchmark。
